# grep / rg を打つ前のチェック

「X が grep にヒットしない」を「X は存在しない」の根拠にしない。
編集の前に利用箇所を grep していないのに「副作用は無い」と判断しない。
grep が証明できるのは検索範囲内のテキスト中の文字列の有無だけ。

## まず最優先で (全 grep 共通)

1. **検索範囲は想定通りか** — どのディレクトリ?  archive (gz / zip) は対象?  binary 除外?  pipe 上流で `head` / `tail` / `--max-count` 等で先に絞られていないか?  stderr が混じる場合は `2>&1` してから grep してるか?

2. **「無い」の結論は表記揺れを試してから** — case 違い / 区切り違い (`-` vs `_` vs space) / encoding / 別 path 階層 / 改行混じりで切れていないか

3. **何かに手を入れる前に「利用箇所」を逆方向で grep** — 提供側 (定義・宣言・fixture・データ・config・設定値など) を編集 / 検証する前に、それを使っている側を必ず洗う:
   - 「使えるか」(存在確認) — 利用例 1 件で確定
   - 「変えたい」(影響範囲確認) — 利用箇所すべてを洗い出してから編集
   - 「消して良いか」(未使用確認) — 利用 0 件を確認

   片方向だけ見て手を入れない。

## コードを grep する時の追加盲点

- **間接定義 / 中継**: catch-all 系の一括公開 / re-export / autoload / 動的読み込み — 個別 symbol 名が grep 対象テキストに現れない
- **生成・変換物**: マクロ展開、コード生成、トランスパイル、minify、別言語へのコンパイル成果物 — 元の symbol 名が消えている
- **入口ファイルを `head` で先に俯瞰** — index / mod / __init__ / lib 等の barrel は先頭数行に catch-all 系の export を持つことが多い。個別 keyword の grep より先に総体を読む

## 結論 / 編集の前のチェックリスト

- [ ] 検索範囲・除外 (`--include` / 再帰 / archive) の設定が広すぎ / 狭すぎないか確認した
- [ ] 利用箇所 (consumer 側) からも grep した
- [ ] 編集する場合: 利用箇所すべての挙動と互換維持を確認した
- [ ] 表記揺れ・別 path を試した
