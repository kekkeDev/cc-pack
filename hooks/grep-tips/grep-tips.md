# grep / rg を打つ前のチェック

「X が grep にヒットしない」を「X は存在しない」の根拠にしない。
grep が証明できるのは検索範囲内のテキスト中の文字列の有無だけ。

## まず最優先で

1. **入口ファイルを `head` で先に俯瞰** — 公開 API の入り口 (index / mod / __init__ / lib など) は先頭数行に catch-all 系の一括公開や include を持つことが多い。個別 keyword の grep より、まずファイル先頭の総体を読む
2. **利用箇所を逆方向で探す** — 「X が使えるか」を知りたい時は、X を定義側に探すより、X を使っている / 参照している箇所を 1 件でも見つけるのが最も確実な存在証拠

## grep が見落とす盲点

- **間接定義 / 中継**: catch-all 系の一括公開 / re-export / autoload / 動的読み込み — 個別 symbol 名が grep 対象テキストに現れない
- **生成・変換**: マクロ展開、コード生成、トランスパイル、minify、別言語へのコンパイル成果物
- **対象外**: 検索範囲に含めていないディレクトリ、バイナリ、シンボリックリンク先、.gitignore 配下、別パッケージ / 別 layer
- **表記揺れ**: case 違い、語形違い、別 path 階層、別エンコーディング

## 否定の結論を出す前のチェックリスト

- [ ] 入口ファイル先頭を `head` で一度読んだ
- [ ] 利用箇所 (consumer 側) からも grep した
- [ ] 表記揺れ・別 path を試した
- [ ] 検索対象範囲 (`--include` / 除外) が広すぎ / 狭すぎないか確認した
